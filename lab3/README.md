# Лабораторная работа 3 (Prolog)

## Тема
Поиск решения в пространстве состояний для головоломки с перестановкой шаров.

## Постановка задачи
Есть линейка из 7 позиций:
- 3 белых шара `w`
- 3 черных шара `b`
- 1 пустая позиция `e`

Начальное состояние:
`[w,w,w,e,b,b,b]`

Целевое состояние:
`[b,b,b,e,w,w,w]`

Нужно найти последовательность допустимых ходов, переводящую старт в цель.

Файл с решением: `lab3/lab3.pl`

## Формализация в Prolog
Состояние кодируется списком длины 7.

- `w` и `b` - шары двух цветов
- `e` - пустая клетка

Это задано фактами:

```prolog
start([w,w,w,e,b,b,b]).
goal([b,b,b,e,w,w,w]).
empty(e).
```

## Правила ходов
Разрешены 2 типа хода:
1. Сдвиг на соседнюю пустую позицию (расстояние 1).
2. Прыжок через один шар в пустую позицию (расстояние 2), если между позициями действительно стоит шар (не `e`).

### Как это выражено в коде
- `adjacent(To, From)` - соседние позиции.
- `jump(To, From, State)` - прыжок через одну занятую позицию.
- `move(State, Next, move(From, To))` - общий оператор перехода между состояниями.

`move/3` делает всё нужное:
- находит индекс пустой клетки `To`;
- проверяет, что из `From` в `To` можно сходить (`adjacent` или `jump`);
- проверяет границы (`within/2`);
- меняет элементы местами (`swap_positions/4`), получая `Next`.

## Вспомогательные предикаты
- `within(State, I)` - индекс `I` в пределах длины списка.
- `replace_nth1/4` - замена элемента по индексу.
- `swap_positions/4` - обмен двух позиций в списке.

Они нужны, чтобы корректно и чисто строить новое состояние без императивных операций.

## Реализованные методы поиска

### 1) DFS (поиск в глубину)
Точка входа: `dfs_path(Path, Moves)`.

Логика:
- углубляемся по первому найденному ходу;
- при откате пробуем альтернативы (backtracking);
- защита от циклов: `\+ member(Next, States)`.

Плюсы:
- простая реализация;
- обычно меньше памяти, чем BFS.

Минусы:
- не гарантирует кратчайший путь;
- может уйти в длинную ветку.

### 2) BFS (поиск в ширину)
Точка входа: `bfs_path(Path, Moves)`.

Логика:
- используется очередь узлов `node(States, Moves)`;
- сначала рассматриваются все состояния на глубине `d`, потом `d+1`.

Теоретически BFS должен находить кратчайший путь по числу ходов.

### 3) IDS (итеративное углубление)
Точка входа: `ids_path(Path, Moves)`.

Логика:
- запускает поиск с лимитом глубины: 0, 1, 2, ... (`between(0, 50, Depth)`);
- на каждом лимите работает `dls/7` (Depth-Limited Search);
- как только цель найдена, поиск завершается.

Плюсы:
- как DFS, использует мало памяти;
- как BFS, находит решение минимальной глубины (при корректной реализации).

## Важная особенность текущей реализации BFS
В `bfs_queue/4` нет глобального множества посещенных состояний (closed set), есть только проверка «не встречалось в текущем пути».

Из-за этого одинаковые состояния могут многократно попадать в очередь через разные пути, что резко увеличивает время работы.

Итог на практике:
- `BFS` в данной версии может работать очень долго.

Это корректно озвучивать на защите как ограничение именно реализации, а не идеи BFS.

### BFS
Запрос:
```prolog
?- bfs_path(Path, Moves).
?- bfs_path(_, Moves), length(Moves, L).
```

В текущем коде запрос может выполняться очень долго из-за повторной генерации уже встречавшихся состояний (см. раздел выше).
То же относится и к `bfs_len(L)`, так как он вызывает `bfs_path/2`.

## Что говорить преподавателю на защите
1. Я представила задачу как поиск в пространстве состояний, где состояние - список из 7 символов.
2. Переход между состояниями полностью описан декларативно предикатом `move/3`.
3. Ограничения хода (соседний шаг/прыжок) выделены отдельно в `adjacent/2` и `jump/3`.
4. Во всех поисках есть защита от циклов на текущем пути: `\+ member(Next, States)`.
5. DFS, BFS и IDS отличаются структурой frontier:
   - DFS: стек (рекурсия)
   - BFS: очередь
   - IDS: серия DFS с растущим лимитом глубины
6. По эксперименту IDS нашел решение за 15 шагов, DFS дал значительно более длинный путь (77).
7. В моей реализации BFS теоретически правильный, но технически не оптимизирован: без глобального visited возникают дубликаты состояний.

## Возможное улучшение (если спросят)
Добавить для BFS глобальный список посещенных состояний (closed set), чтобы каждое состояние обрабатывалось один раз. Тогда BFS стабильно даст кратчайший путь и будет работать значительно быстрее.
